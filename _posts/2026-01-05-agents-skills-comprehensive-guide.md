# はじめに
この記事では、AIエージェント開発の二大潮流「AGENTS SKILLS」と「MCP（Model Context Protocol）」について、その基本から実践的な活用例まで、2026年最新の視点で徹底解説します。両技術の違いと互补関係を理解し、次世代AI開発の道筋を描きます。

**対象読者**: AIエージェント開発者、技術アーキテクチャの検討者、AI活用を検討しているビジネス担当者

# 1. 両技術の基本定義

## MCP（Model Context Protocol）とは？
**「AIモデルと外部システムを標準化された方法で接続するオープンプロトコル」**

*   **発表**: 2024年11月、Anthropic初発表
*   **目的**: AIモデルが外部データソースやツールに安全にアクセスするための共通プロトコル
*   **位置付け**: 「通信層」「接続層」
*   **特徴**: JSON-RPC 2.0ベースのプロトコル

```
┌─────────────┐         MCP           ┌──────────────┐
│  AIモデル   │ ◄───────締───────►  │ 外部データ   │
│  (Claude)   │  標準化された接続    │  (DB, API等) │
└─────────────┘                      └──────────────┘
```

## AGENTS SKILLSとは？
**「AIエージェントの能力をモジュール化し、再利用可能にするためのスキル・アーキテクチャ」**

*   **発表**: 2025年11月、Anthropicが公開
*   **目的**: 汎用AIエージェントを特定業務特化型へ進化させる
*   **位置付け**: 「知識層」「行動層」
*   **特徴**: SKILL.md形式での定義、独立した機能モジュール

```
┌─────────────────┐
│ AGENTS SKILLS   │
├─────────────────┤
│ ・google-search │
│ ・curl          │
│ ・cron-schedule │
│ ・blog-know-to  │
└─────────────────┘
       ↓
┌─────────────────┐
│ 組み合わせ     │
│ 特化エージェント│
└─────────────────┘
```

# 2. なぜ今AGENTS SKILLSが注目されているのか？

## AGENTS SKILLSの実例から見る価値
まずは、私たちのプロジェクトで実際に実装済みのスキルを見てみましょう：

| スキル名 | 機能概要 | 主な利用场景 |
|----------|----------|--------------|
| **google-search** | Google検索実行 | 情報収集、最新トレンド調査 |
| **duckduckgo-search** | DuckDuckGo検索 | プライバシー重視の検索 |
| **curl** | HTTPリクエスト送信 | API連携、Webデータ取得 |
| **cron-scheduler** | ジョブスケジュール管理 | 定期実行タスクの自動化 |
| **blog-know-to** | ブログ作成・公開 | 知識発信、アウトプット自動化 |
| **fetch-news** | ニュース取得 | リアルタイム情報収集 |

## 既存AIアーゕテクチャの課題

### モノリシック設計の問題点
```
従来: エージェント（全機能内蔵）
     ↓
AGENTS SKILLS: エージェント（フレームワーク） + スキル（機能モジュール）
```

| 課題 | 従来方式 | AGENTS SKILLSの解決策 |
|------|----------|----------------------|
| 機能追加 | コード修正が必要 | 新しいスキル追加のみ |
| カスタマイズ | 全機能持つ | 必要なスキルのみ組合せ |
| テスト | 全体テスト必要 | スキル単位で独立テスト |
| 共有 | エージェントごと | 同じスキルを複数で共有 |

# 3. MCPとの技術的差異比較

## 両技術の比較表

| 項目 | MCP | AGENTS SKILLS |
|------|-----|---------------|
| **目的** | 外部接続の標準化 | 能力のモジュール化・再利用 |
| **位置付け** | プロトコル（通信仕様） | アーキテクチャ（設計思想） |
| **粒度** | ツール/API単位 | 業務レベルのスキル単位 |
| **実装形態** | JSON-RPCプロトコル | SKILL.md + 実装コード |
| **主要機能** | データアクセス・ツール実行 | 複数ステップの処理フロー |
| **再利用性** | 接続先の再利用 | スキル自体の再利用 |
| **公開形態** | サーバー/クライアント | スキルパッケージ |
| **主な用途** | 外部データ取得 | 複雑業務の自動化 |

## 課題解決のアプローチの違い

### MCPの解決課題
**「AIモデルと外部ツールの接続混沌」**

**従来の課題例**: 
*   Gmail連携 → Gmail API学習 + コード実装
*   Slack連携 → Slack API学習 + コード実装
*   DB接続   → DBドライバ学習 + コード実装

**MCPの解決**: 
*   MCPクライアント実装
*   あらゆるMCP対応サーバーに接続可能
*   標準化された接続方式

### AGENTS SKILLSの解決課題
**「汎用AIの業務適用の壁」**

**従来の課題例**:
```
汎用AI: 「社内報を書いて、Slackに投稿して」
        → 手順・格式・タイミングが人により異なる
        → 結果の再現性・再利用性が低い
```

**AGENTS SKILLSの解決**:
```
AGENTS SKILLS: 
  1. fetch-newsで情報収集 → 結果形式が固定
  2. LLMで社内報作成     → テンプレート固定
  3. curlでSlack投稿     → 送信形式固定
  → 誰でも同じ結果が再現
```

# 4. 技術的実装の比較

## MCPの実装例
```python
# MCPクライアント設定
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/dir"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {"GITHUB_TOKEN": "xxx"}
    }
  }
}

# 使用時
Claude → "GitHubの最新リポジトリを取得して"
       → MCPクライアント → GitHub MCPサーバー → データ取得
```

## AGENTS SKILLSの実装例
```python
# SKILL.md定義
---
name: 社内報作成スキル
description: ニュースを収集し、社内報形式でSlack投稿
---

1. fetch-newsで業界ニュース取得
2. LLMで要約・社内報フォーマット
3. curlでSlack投稿
4. 結果をlog出力

# 使用時
Agent Skills → "社内報作成スキルを実行"
             → 自動的に上記4ステップ実行
```

# 5. 共存の可能性と実践的活用

## 両技術の組み合わせパターン

### パターン1: MCPで接続 → SKILLSで活用
```
外部データソース ←MCP→ AIエージェント ←SKILLS→ 業務自動化
          ↓                    ↓
     標準化接続          結果を業務レベルスキルとして利用
```

**例**: 社内DB（MCP接続）+ 報告作成スキル

### パターン2: SKILLS内でのMCP活用
```
Skills: 市場調査スキル
├─ MCP: 外部APIからデータ取得
├─ MCP: ニュースソース接続
└─ 内部処理: 分析・要約・報告
```

**例**: 競合分析スキルが複数MCPソースを活用

### パターン3: 異なる層での共存
| 層 | 担当 | 技術 |
|----|------|------|
| 接続層 | 外部データアクセス | MCP |
| 知識層 | 業務手順・判断基準 | AGENTS SKILLS |
| 実行層 | 処理実行・制御 | 両方 |

## 具体的な活用事例

### 事例1：自動研究支援エージェント
スキル組み合わせ例：`google-search` + `curl` + `blog-know-to`

**Scenario**: 学術論文の自動要約・公開システム

1.  **情報収集**: `google-search`で最新研究を検索
2.  **データ取得**: `curl`で論文PDFを取得
3.  **処理**: 要約生成（LLM標準機能）
4.  **公開**: `blog-know-to`でブログ記事として公開

### 事例2：業務自動化アシスタント
スキル組み合わせ例：`cron-scheduler` + `fetch-news` + `curl`

**Scenario**: 競合動向の定時レポート作成

*   **08:00**: cron-schedulerで起動
*   **fetch-news**: ニュース取得
*   **curl**: 各社のRSS/APIからデータ収集
*   **LLM**: レポート生成
*   **email**: 結果送信（送信スキル追加予定）

### 事例3：カスタマイズ型カスタマーサポート
スキル組み合わせ例：`google-search` + `curl` + 業務特化スキル

**Scenario**: 製品サポート専門エージェント

1.  製品情報を検索（google-search）
2.  在庫状況を確認（curl + 内部API）
3.  解決策を提案（LLM + ドキュメント検索）
4.  対応履歴を記録（curl + CRM API）

# 6. 導入の判断基準

## MCPを優先すべきケース
*   **「既存の多数のツール/APIを統一的に接続したい」**
    *   外部システムとの多様な連携が必要
    *   統一インターフェースで接続したい
    *   データアクセス層の標準化
    *   **具体的には**: 複数のSaaS連携、DB接続、API統合

## AGENTS SKILLSを優先すべきケース
*   **「特定業務を自動化・再現したい」**
    *   定型業務の自動化
    *   複数ステップを含む処理
    *   スキルの共有・再利用
    *   **具体的には**: 報告作成、データ分析、顧客対応

## 両方使うべきケース
*   **「多様なデータを活用して複雑業務を自動化」**
    *   外部API・DBからデータ収集（MCP）
    *   収集データを元に業務処理（SKILLS）
    *   **具体的には**: 市場調査・分析自動化、競合分析システム

# 7. 技術的アーキテクチャ

## AGENTS SKILLSの標準実装パターン
```python
# 基本構造
def skill_function(input_params):
    # 入力処理
    # メイン処理
    # 出力変換
    return result
```

## スキル連携の仕組み
```
ユーザー要求
    ↓
LLM（ルーティング判断）
    ↓
必要なスキル選択
    ↓
スキル実行
    ↓
結果統合
    ↓
最終出力
```

## エラーハンドリング
各スキルは独立しているため、エラー発生時も影響範囲を最小限に抑えることができます。
