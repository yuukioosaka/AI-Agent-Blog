# はじめに
この記事では、AIエージェント開発の二大潮流「AGENTS SKILLS」と「MCP（Model Context Protocol）」の違いを深掘りします。両者が競合するのか、それとも共存するのか。実際の技術的特性から実践的な活用例まで、2026年最新の視点で解説します。

**対象読者**: AIエージェント開発者、技術アーキテクチャの検討者、両技術の比較をしたい方

# 1. 両技術の基本定義

## MCP（Model Context Protocol）とは？
**「AIモデルと外部システムを標準化された方法で接続するオープンプロトコル」**

*   **発表**: 2024年11月、Anthropic初発表
*   **目的**: AIモデルが外部データソースやツールに安全にアクセスするための共通プロトコル
*   **位置付け**: 「通信層」「接続層」
*   **特徴**: JSON-RPC 2.0ベースのプロトコル

```
┌─────────────┐         MCP           ┌──────────────┐
│  AIモデル   │ ◄───────締───────►  │ 外部データ   │
│  (Claude)   │  標準化された接続    │  (DB, API等) │
└─────────────┘                      └──────────────┘
```

## AGENTS SKILLSとは？
**「AIエージェントの能力をモジュール化し、再利用可能にするためのスキル・アーキテクチャ」**

*   **発表**: 2025年11月、Anthropicが公開
*   **目的**: 汎用AIエージェントを特定業務特化型へ進化させる
*   **位置付け**: 「知識層」「行動層」
*   **特徴**: SKILL.md形式での定義、独立した機能モジュール

```
┌─────────────────┐
│ AGENTS SKILLS   │
├─────────────────┤
│ ・google-search │
│ ・curl          │
│ ・cron-schedule │
│ ・blog-know-to  │
└─────────────────┘
       ↓
┌─────────────────┐
│ 組み合わせ     │
│ 特化エージェント│
└─────────────────┘
```

# 2. 技術的差異の比較表

| 項目 | MCP | AGENTS SKILLS |
|------|-----|---------------|
| **目的** | 外部接続の標準化 | 能力のモジュール化・再利用 |
| **位置付け** | プロトコル（通信仕様） | アーキテクチャ（設計思想） |
| **粒度** | ツール/API単位 | 業務レベルのスキル単位 |
| **実装形態** | JSON-RPCプロトコル | SKILL.md + 実装コード |
| **主要機能** | データアクセス・ツール実行 | 複数ステップの処理フロー |
| **再利用性** | 接続先の再利用 | スキル自体の再利用 |
| **公開形態** | サーバー/クライアント | スキルパッケージ |
| **主な用途** | 外部データ取得 | 複雑業務の自動化 |

# 3. なぜこの違いが重要なのか？

## 課題解決のアプローチの違い

### MCPの解決課題
**「AIモデルと外部ツールの接続混沌」**
*   ツールごとに個別のAPI連携が必要
*   接続コードの重複と保守コスト
*   セキュリティ・認証の不統一

**例**: 
```
従来: Gmail連携 → Gmail API学習 + コード実装
      Slack連携 → Slack API学習 + コード実装
      DB接続   → DBドライバ学習 + コード実装

MCP: MCPクライアント実装 → あらゆるMCP対CP対応サーバーに接続可能
```

### AGENTS SKILLSの解決課題
**「汎用AIの業務適用の壁」**
*   汎用AIでは業務フローが不明確
*   プロンプトだけで再現性・再利用性が低い
*   複雑業務の自動化に実装コストが高い

**例**:
```
汎用AI: 「社内報を書いて、Slackに投稿して」
        → 手順・格式・タイミングが人により異なる

AGENTS SKILLS: 
  1. fetch-newsで情報収集 → 結果形式が固定
  2. LLMで社内報作成     → テンプレート固定
  3. curlでSlack投稿     → 送信形式固定
  → 誰でも同じ結果が再現
```

# 4. 実際のコード比較

## MCPの実装例
```python
# MCPクライアント設定
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/dir"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {"GITHUB_TOKEN": "xxx"}
    }
  }
}

# 使用時
Claude → "GitHubの最新リポジトリを取得して"
       → MCPクライアント → GitHub MCPサーバー → データ取得
```

## AGENTS SKILLSの実装例
```python
# SKILL.md定義
---
name: 社内報作成スキル
description: ニュースを収集し、社内報形式でSlack投稿
---

1. fetch-newsで業界ニュース取得
2. LLMで要約・社内報フォーマット
3. curlでSlack投稿
4. 結果をlog出力

# 使用時
Agent Skills → "社内報作成スキルを実行"
             → 自動的に上記4ステップ実行
```

# 5. 共存の可能性と実践的活用

## 両技術の組み合わせパターン

### パターン1: MCPで接続 → SKILLSで活用
```
外部データソース ←MCP→ AIエージェント ←SKILLS→ 業務自動化
          ↓                    ↓
     標準化接続          結果を業務レベルスキルとして利用
```

**例**: 社内DB（MCP接続）+ 報告作成スキル

### パターン2: SKILLS内でのMCP活用
```
Skills: 市場調査スキル
├─ MCP: 外部APIからデータ取得
├─ MCP: ニュースソース接続
└─ 内部処理: 分析・要約・報告
```

**例**: 競合分析スキルが複数MCPソースを活用

### パターン3: 異なる層での共存
| 層 | 担当 | 技術 |
|----|------|------|
| 接続層 | 外部データアクセス | MCP |
| 知識層 | 業務手順・判断基準 | AGENTS SKILLS |
| 実行層 | 処理実行・制御 | 両方 |

# 6. 導入の判断基準

## MCPを優先すべきケース
*   **「既存の多数のツール/APIを統一的に接続したい」**
    *   外部システムとの多様な連携が必要
    *   統一インターフェースで接続したい
    *   データアクセス層の標準化

## AGENTS SKILLSを優先すべきケース
*   **「特定業務を自動化・再現したい」**
    *   定型業務の自動化
    *   複数ステップを含む処理
    *   スキルの共有・再利用

## 両方使うべきケース
*   **「多様なデータを活用して複雑業務を自動化」**
    *   外部API・DBからデータ収集（MCP）
    *   収集データを元に業務処理（SKILLS）
